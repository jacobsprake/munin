# Munin Sovereign Orchestration Packet — Format Specification v1

## Status: DRAFT

## Abstract

This document specifies the wire format for Munin Authorization Packets (MAPs). A MAP is a self-contained, cryptographically signed data structure that carries an infrastructure incident analysis, recommended response playbook, supporting evidence, regulatory basis, and multi-party authorization chain. MAPs are designed to be defensible audit records that survive the operational lifetime of the infrastructure they protect.

## 1. Terminology

- **MAP**: Munin Authorization Packet
- **Shadow Link**: A dependency between infrastructure assets that exists in physics but is not documented in any asset registry
- **Evidence Window**: A time-bounded observation period supporting a causal inference
- **Quorum**: The minimum number of authorized signatories required to approve a MAP

## 2. Packet Structure

```json
{
  "version": "1.0",
  "id": "<UUID>",
  "createdAt": "<ISO 8601 timestamp>",
  "status": "<draft|ready|authorized|executed|verified|closed>",

  "scope": {
    "regions": ["<region_id>"],
    "nodeIds": ["<node_id>"],
    "services": ["<service_type>"]
  },

  "situationSummary": "<human-readable incident description>",

  "proposedAction": {
    "playbookId": "<playbook_filename>",
    "actions": [
      {
        "step": 1,
        "description": "<action description>",
        "target": "<node_id>",
        "type": "<advisory|notification|coordination>"
      }
    ]
  },

  "evidenceRefs": ["<evidence_window_id>"],

  "uncertainty": {
    "confidence": 0.0-1.0,
    "counterexampleWindows": ["<window where prediction didn't hold>"],
    "confounderNotes": ["<potential confounding factors>"]
  },

  "regulatoryBasis": ["<statute_reference>"],

  "multiSig": {
    "required": "<number of required signatures>",
    "threshold": "<minimum signatures for authorization>",
    "currentSignatures": 0,
    "signatures": [
      {
        "signerId": "<ministry_or_operator_id>",
        "role": "<rbac_role>",

        "timestamp": "<ISO 8601>"
      }
    ],
    "minimumSignOff": false,
    "minimumSignOffRole": null
  },

  "pqcSignature": {
    "algorithm": "ML-DSA-65",
    "signature": "<base64-encoded PQC signature>",

  },

```

## 3. Status Transitions

```
draft --> ready --> authorized --> executed --> verified --> closed
  |                    |
  +---> (rejected)     +---> (expired)
```

- `draft`: Packet generated by engine, not yet reviewed
- `ready`: Packet reviewed by system, evidence complete
- `authorized`: Required quorum of signatures obtained
- `executed`: Recommended actions taken by human operators
- `verified`: Post-execution verification confirms outcomes
- `closed`: Incident resolved, packet archived

## 4. Signature Requirements

### Classical Signature (Required)
Every MAP must include at least one Ed25519 signature from an authorized operator or ministry representative.

### Post-Quantum Signature (Recommended)
When PQC is enabled, the MAP also includes an ML-DSA-65 signature. Both classical and PQC signatures must verify for the packet to be considered valid.

### Multi-Signature Quorum
The `multiSig.threshold` field specifies the minimum number of signatures required. Different deployment profiles use different thresholds:
- Demo: 1-of-1
- Pilot: 2-of-3
- Production: 3-of-4
- Emergency: Reduced threshold with enhanced audit logging

## 5. Merkle Chain Linkage

Each MAP includes a Merkle receipt that chains it to the previous packet. This creates a tamper-evident, append-only log of all authorization actions. Chain integrity can be verified independently of the signing authority.

## 6. Security Considerations

- MAPs must be stored in an air-gapped environment
- MAPs must not be transmitted over untrusted networks
- Private signing keys must be stored in HSMs or TEE enclaves
- Key rotation must be tracked in the user_key_history table
- Expired or revoked keys must not be accepted for new signatures

## 7. Validation Rules

A MAP is valid if and only if:
1. All required fields are present and correctly typed
2. Status transition is valid (per Section 3)
3. At least one evidence reference exists
4. Regulatory basis is non-empty
5. Confidence score is between 0.0 and 1.0
6. Multi-sig quorum is met (currentSignatures >= threshold)
7. Merkle receipt chain is intact
8. All signatures verify against registered public keys

## 8. Schema Reference

The JSON Schema for MAPs is maintained at `contracts/packet.schema.json`.

## 9. Versioning

The `version` field follows semantic versioning. Breaking changes increment the major version. The current version is 1.0.
