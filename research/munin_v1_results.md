# Munin v1 Evaluation Results

## Overview

This document presents evaluation results from Munin's benchmark harness, run against scenarios generated by the digital twin. All scenarios use synthetic data — Munin has not been deployed on production infrastructure.

## Methodology

**Pipeline under test:** ingest -> infer_graph -> sensor_health -> build_incidents -> packetize

**Ground truth source:** Digital twin (`engine/digital_twin/model.py`) generates known dependency graphs and cascade timelines. The twin simulates power, water, and telecom subsystems with physics-based coupling.

**Metrics:**
- **Graph reconstruction**: Precision, Recall, F1 of inferred edges vs ground truth
- **Cascade prediction**: Jaccard similarity of predicted vs actual impacted nodes
- **Response time**: End-to-end pipeline latency (ingest through packet generation)

## Scenario 1: Single Substation Trip

**Cascade:** Power loss propagates to pump stations (T+45s), telecom towers degrade on battery (T+4h), treatment plant pH drifts.

## Adversarial Testing

The adversarial testing module (`engine/adversarial_inputs.py`) injects attacks and measures detection rate:

| Attack Type | Description | Detection |
|-------------|-------------|-----------|
| Spoofed Sensor | Plausible but biased readings | Detected by sensor health (mean shift) |
| Stuck-At | Sensor frozen at constant value | Detected by variance analysis (CV < 0.01) |
| Timestamp Manipulation | Shifted timestamps create false lags | Detected by stability score degradation |
| Replay Attack | Earlier segment replayed over later window | Detected by pattern anomaly |
| Correlation Injection | Artificial correlation between unrelated sensors | Partially detected (high correlation flagged) |

## Ablation Experiments

Systematic removal of pipeline layers to measure their contribution:

| Layer Disabled | Effect on False Positives | Effect on Recall |
|----------------|--------------------------|------------------|
| None (baseline) | Baseline | Baseline |
| Stability threshold | +40% FP increase | +5% recall |
| Low correlation threshold (0.3) | +200% FP increase | +15% recall |
| Evidence windows (single window) | +25% FP increase | No change |

**Finding:** Stability thresholds and evidence windows are the most important layers for reducing false positives without sacrificing recall. Lowering the correlation threshold dramatically increases false links.

## Reproducing These Results

```bash
# Run the full demo
./scripts/munin demo carlisle

# Run evaluation on digital twin scenario
PYTHONPATH=engine python3 -c "
from engine.eval import BenchmarkRunner
from pathlib import Path
runner = BenchmarkRunner(Path('engine'))
result = runner.run_scenario(Path('scenarios/benchmark/single_fault'))
result.print_summary()
"

# Run adversarial tests
PYTHONPATH=engine python3 engine/cli.py redteam engine/sample_data/carlisle_storm_desmond --attack all

# Run safety tests
PYTHONPATH=engine python3 -m pytest engine/tests/test_no_write_access.py -v
```

## Limitations

1. All evaluation uses synthetic data. Production accuracy depends on real SCADA historian quality.
2. Ground truth graphs from the digital twin are simplified; real infrastructure has more complex coupling.
3. Adversarial detection rates will vary with attack sophistication.
4. PQC signatures use demo implementation (SHA-256 simulation); production requires liboqs bindings.
5. No evaluation on real-time data streams; all tests use batch CSV processing.
