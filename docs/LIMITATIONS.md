# Limitations and Open Problems

What Munin can't do yet, and the hardest problems I see. For each, how I plan to attack it.

---

## Technical limitations

**Correlation is not causation.**
Shadow-link discovery uses temporal correlation with lag detection. This finds *candidate* dependencies, not proven causal relationships. Confounders (weather, shared maintenance schedules, demand cycles) can produce spurious links. Munin mitigates this with stability scoring, confounder notes, evidence windows, and counterexample tracking — but the fundamental limitation remains. Attack plan: integrate physics-based constraints from domain models to reject links that are statistically correlated but physically implausible.

**Synthetic data only.**
All evaluation results are on synthetic data generated by the digital twin, plus public flood datasets. The system has not been tested on real SCADA historian exports from operational infrastructure. Synthetic data, no matter how realistic, cannot capture the full messiness of real-world sensor networks: intermittent connectivity, vendor-specific protocol quirks, undocumented data formats, and the long-tail of failure modes that only appear in production. Attack plan: secure pilot access to real historian data (see WHATS_NEXT.md).

**PQC is simulated.**
The ML-DSA (Dilithium-3) signatures use SHA-256 simulation, not actual lattice-based cryptography. This means the system is not quantum-resistant today — it has the API surface and dual-stack architecture for PQC, but not the actual cryptographic strength. Attack plan: integrate liboqs bindings. The engineering is straightforward; the API is already designed for it.

**TEE is software fallback.**
The Trusted Execution Environment attestation runs in `SOFTWARE_FALLBACK` mode. Logic-Lock constraints are enforced in regular process memory, not in a hardware enclave. A compromised operating system could theoretically bypass them. Attack plan: SGX/SEV hardware integration (requires development boards).

**No real-time streaming.**
The engine processes batch CSV data, not real-time SCADA streams. In a production deployment, you'd need sub-second ingestion from OPC UA or Modbus TCP. The architecture supports this (the connector framework exists), but it hasn't been tested under real-time load with backpressure handling. Attack plan: build a streaming ingestion layer with configurable buffer and backpressure.

**Single-geography only.**
The system has been designed and tested for one geography (Carlisle, UK). Scaling to national or multi-national scope introduces challenges: graph size, latency across distributed deployments, federated governance across jurisdictions, and regulatory harmonization. Attack plan: this is a scaling problem, not an architecture problem — the design is sector-agnostic and jurisdiction-parameterized.

## Institutional limitations

**No production deployment.**
Munin has never run on real infrastructure. The gap between "works on my laptop" and "deployed in a government secure facility" is enormous: security accreditation, operational handover, training, maintenance contracts, and institutional buy-in. This is a multi-year process for any infrastructure tool, and Munin is at step zero. Attack plan: start with shadow-mode pilots where Munin runs alongside existing tools with zero risk.

**No formal safety accreditation.**
The safety case and STPA analysis are internally produced. They have not been reviewed by an independent safety assessor, accredited against IEC 62443, or accepted by any regulatory body. In critical infrastructure, internal safety cases carry very little weight. Attack plan: engage an independent assessor for a lightweight review, then pursue formal accreditation if a pilot partner requires it.

**Single-person development.**
This is built by one person. That means: no code review beyond automated linting, no independent security audit, no diversity of perspectives in design decisions, and a bus factor of one. Attack plan: this is the most honest limitation and the hardest to fix without funding. Priority #1 for any resources is bringing in at least one other engineer with ICS/OT experience.

## Open research questions

**How do you calibrate confidence scores?**
Munin assigns confidence scores to edges and packets, but there's no ground truth for calibration. A "0.85 confidence" edge doesn't mean "85% chance this is a real dependency" in any rigorous probabilistic sense. It means "the correlation is 0.85 and the stability score is above threshold." Making these scores meaningful requires calibration against known dependency graphs from real infrastructure — which we don't have yet.

**How do you handle adversarial operators?**
M-of-N signing prevents a single compromised ministry from authorizing dangerous actions. But what if M ministries are compromised? The quorum threshold is a governance parameter, not a security proof. Byzantine fault tolerance has known limits: it requires honest majority. This is a fundamental limitation of any governance system, not specific to Munin, but it's important to be explicit about it.

**What's the right level of automation?**
Munin is advisory-only. But where's the line? If an operator approves every Munin recommendation without reading the evidence, the system has effectively become autonomous — the human is just a rubber stamp. This is the automation bias problem, and it's not solved by architecture alone. It requires training, UI design that forces engagement, and institutional culture that values independent judgment. We've thought about this (see HUMAN_FACTORS.md) but haven't solved it.

**Can this survive adversarial procurement?**
A government buyer who wants to sabotage Munin (e.g., because it threatens their existing vendor relationship) can find reasons to reject it. The RFP pack and standards mapping exist to make this harder, but ultimately Munin needs a champion inside the buying organization. Finding and supporting that champion is a go-to-market problem, not a technical one.
