# UX Rationale: Authorisation Console

**Document ID:** UX-RAT-001 v1.0  
**Classification:** OFFICIAL  
**Date:** 2026-03-01  
**Owner:** Product Design Unit

---

## 1. Design Philosophy

The Authorisation Console (`/handshakes`, `components/HandshakePanel.tsx`, `components/PacketTable.tsx`) is the primary interface through which operators review and approve Munin-generated authorisation packets. Every UI element in the console exists to address a specific cognitive failure mode documented in the human factors literature for decision-support systems in high-stakes environments (aviation, nuclear, emergency management).

The design follows three principles:

1. **Evidence-first, not recommendation-first.** The operator sees the evidence before the recommendation. This prevents automation bias (uncritical acceptance of system output).
2. **Uncertainty as a first-class citizen.** Confidence is never hidden. The operator always knows how certain the system is and why.
3. **Friction by design.** The approval workflow introduces deliberate friction at decision points where rubber-stamping is most dangerous.

## 2. UI Element Rationale

### 2.1 Uncertainty Display

**Component:** Rendered from `uncertainty.overall` and `uncertainty.notes` fields (`lib/packet/types.ts` — `UncertaintyBlock`).

#### Failure Mode Addressed: Automation Bias

**Description:** Automation bias is the tendency for operators to accept system recommendations without critical evaluation. In decision-support systems, operators treat high-confidence outputs as instructions and low-confidence outputs as noise, ignoring the nuance in between. Research on clinical decision support (Goddard et al., 2012) and aviation autopilot reliance (Parasuraman & Riley, 1997) shows that numerical probability displays (e.g., "87.3% confident") are particularly susceptible to anchoring effects — operators latch onto the number and stop thinking.

#### Design Decision: Ordinal Labels + Raw Probability + Explanatory Notes

The uncertainty display uses a three-layer approach:

1. **Ordinal label (HIGH / MEDIUM / LOW):** Displayed prominently as a colour-coded badge.
   - HIGH (green, `>= 0.8`): Strong evidence, stable correlations.
   - MEDIUM (amber, `0.5–0.8`): Mixed evidence, operator scrutiny required.
   - LOW (red, `< 0.5`): Weak evidence, additional investigation needed.
   
   *Rationale:* Ordinal labels force categorical thinking. An operator who sees "MEDIUM" engages differently than one who sees "0.67" — the label communicates "this requires your judgement" rather than "this is probably fine."

2. **Raw probability (small text below label):** The exact `uncertainty.overall` value (e.g., `0.67`) is displayed in smaller text for operators who want precision.
   
   *Rationale:* Removing the raw number entirely would reduce transparency. Operators must be able to inspect the exact value for audit and discussion purposes.

3. **Uncertainty notes (expandable section):** The `uncertainty.notes[]` array is displayed as a bulleted list below the label. Notes include contextual information like `"Limited historical evidence for this scenario"` and `"Large blast radius increases prediction uncertainty"` (generated by `engine/packetize.py`).
   
   *Rationale:* Notes provide the "why" behind the number. An operator who sees "67% confidence" cannot act on it. An operator who sees "67% — limited historical evidence for this scenario, 3 counterexample windows found" can make an informed decision about whether to seek additional data.

4. **Counterexample count:** If `uncertainty.counterexampleCount` is present (from `lib/packet/types.ts`), it is displayed as a distinct warning badge: `"⚠ 3 counterexample windows"`.
   
   *Rationale:* Counterexamples are the single most important piece of information for preventing automation bias. They represent direct evidence against the recommendation. Making them invisible (or buried in a data table) would undermine the entire trust architecture.

#### Validation Rule Connection

`lib/packet/validate.ts` emits a warning when `uncertainty.overall < 0.5`: `"Low confidence — requires additional evidence"`. This warning is rendered as a persistent banner on the packet detail view, ensuring the operator cannot approve without acknowledging the low confidence.

### 2.2 Evidence Windows List

**Component:** Rendered in the Evidence Panel (`components/EvidencePanel.tsx`), populated from `evidenceRefs[]` and the underlying evidence data (`engine/out/evidence.json`).

#### Failure Mode Addressed: Tunnel Vision (Confirmation Bias)

**Description:** Tunnel vision in crisis response is the tendency to focus on evidence that confirms the initial assessment while ignoring contradictory data. Operators who believe a flood is occurring will selectively attend to rising water-level data while ignoring stable pressure readings that might indicate a sensor malfunction rather than a genuine event. This is exacerbated by time pressure.

#### Design Decision: Dual-Column Layout with Counterexample Highlighting

The evidence windows list is displayed as a dual-column layout:

1. **Supporting windows (left column, green-bordered):** Evidence windows with `supportType: "support"` — positive correlation in the relevant time window. Each window shows:
   - Time range (`startTs` — `endTs`)
   - Correlation value within that window
   - Quality context (missingness, noise, drift scores)
   - Robustness score

2. **Counterexample windows (right column, amber-bordered):** Evidence windows with `supportType: "counterexample"` — negative or weak correlation in the relevant time window. Same data fields as supporting windows but with visual emphasis (amber border, warning icon).

*Rationale:* By presenting counterexamples with equal visual weight and spatial prominence as supporting evidence, the UI forces operators to confront disconfirming data. Hiding counterexamples in a collapsed section or footnote would enable confirmation bias. The dual-column layout makes it physically impossible to review supporting evidence without seeing counterexamples in peripheral vision.

3. **Quality context badges:** Each evidence window displays sensor health indicators (missingness, noise, drift) as small colour-coded badges.
   
   *Rationale:* An evidence window with 98% correlation but 15% missingness is less reliable than one with 85% correlation and 0% missingness. Operators must be able to assess evidence quality at a glance.

4. **Edge provenance link:** Each evidence window links back to the specific edge in the dependency graph (`edgeId`), allowing the operator to click through to the graph view (`/graph`) and see the edge in spatial context.

#### CLI Complement

The `munin evidence-quality` CLI command (`engine/cli.py` — `_run_evidence_quality()`) provides a terminal-based evidence quality dashboard via `engine/analysis/evidence_quality.py` — `EvidenceQualityDashboard`, enabling review without the web UI.

### 2.3 Diff Panel

**Component:** Rendered using `lib/packet/diff.ts` — `diffPackets()`. Displayed when a packet revision exists (i.e., the operator is comparing the current packet against a prior version).

#### Failure Mode Addressed: Over-Reacting or Under-Reacting to Changes

**Description:** When a packet is revised (new evidence added, scope changed, confidence updated), operators face two failure modes:
- **Over-reaction:** A small confidence decrease (e.g., 82% → 78%) causes the operator to reject a still-defensible packet.
- **Under-reaction:** A significant scope expansion (3 nodes → 12 nodes) is overlooked because the operator focuses only on the confidence number.

Both failures stem from the same root cause: operators cannot quickly assess what changed and whether the change matters.

#### Design Decision: Structured Risk-Delta Summary

The diff panel displays a structured `PacketDiff` object (`lib/packet/diff.ts`) with four sections:

1. **Risk Delta:**
   - `confidenceBefore` / `confidenceAfter` with directional indicator (`improved` / `degraded` / `unchanged`).
   - Displayed as a horizontal bar with colour transition (green→amber→red).
   - `constraintsAdded` / `constraintsRemoved` / `failedConstraintsAdded` / `failedConstraintsRemoved` — physics constraint changes.
   
   *Rationale:* Separating confidence change from constraint change prevents conflation. A confidence improvement with a new failed constraint is a dangerous combination that would be missed if only confidence were displayed.

2. **Scope Delta:**
   - `nodesAdded` / `nodesRemoved` — infrastructure nodes entering or leaving the blast radius.
   - `regionsAdded` / `regionsRemoved` — geographic scope changes.
   - Displayed as a tag list with `+` (green) and `−` (red) indicators.
   
   *Rationale:* Scope changes are the highest-impact revisions. Adding 5 nodes to a packet means 5 additional infrastructure assets are predicted to be affected. This must be visually prominent.

3. **Evidence Delta:**
   - `added` / `removed` — evidence reference IDs that were added or removed.
   - Count displayed prominently: `"+3 evidence refs, −1 evidence ref"`.
   
   *Rationale:* Evidence additions strengthen the packet; evidence removals weaken it. The operator needs to know whether the evidence base is growing or shrinking.

4. **Summary (human-readable):**
   - The `summary[]` array provides natural-language descriptions of all changes, generated by `diffPackets()`.
   - Examples: `"Confidence increased from 72% to 85%"`, `"Scope expanded: +pump_station_03, +pump_station_04"`, `"New constraints FAILED: Coordination capacity"`.
   
   *Rationale:* Summary text is the first thing an operator reads. It must be accurate, specific, and actionable — not generic.

### 2.4 Multi-Sig Progress Bar

**Component:** Rendered from `multiSig` block (`lib/packet/types.ts` — `MultiSigBlock`): `currentSignatures`, `threshold`, `required`, and optional `signers[]`.

#### Failure Mode Addressed: Diffusion of Responsibility

**Description:** In multi-agency approval workflows, diffusion of responsibility occurs when each signer assumes another signer will perform due diligence. Classic bystander-effect dynamics apply: the more signers required, the less responsibility each individual feels. In crisis conditions with time pressure, this degenerates into rubber-stamping where each ministry signs because the previous one already did, without independent review.

#### Design Decision: Named Signer Progress with Individual Status Tracking

The multi-sig progress bar displays:

1. **Progress fraction:** `currentSignatures / threshold` displayed as both a fraction (e.g., `"1/3"`) and a fill bar.
   
   *Rationale:* The fill bar creates urgency without panic. A partially filled bar communicates "others have signed; your review is needed" rather than "you are the only one who hasn't signed."

2. **Named signers (when available):** If `signers[]` is populated, each signer's role is displayed with a status indicator:
   - ✓ Signed (green)
   - ○ Pending (grey)
   - ✗ Rejected (red)
   
   *Rationale:* Named attribution prevents anonymity in the approval process. Each signer knows their decision is recorded and visible. This counteracts the bystander effect by making individual responsibility explicit.

3. **Quorum type indicator:** The bar header shows whether this is a standard quorum (`"2 of 3 required"`) or a critical quorum (`"3 of 4 required — all designated ministries must sign"`).
   
   *Rationale:* Operators must understand the consequences of their signature. In a critical quorum, their signature is not optional — the bar explicitly states that all designated ministries must sign.

4. **Time-to-authorise metric:** If `firstApprovalTs` is set (from `engine/packetize.py`), elapsed time since first approval is displayed. This measures coordination latency — the key metric Munin aims to reduce from 2–6 hours to < 30 minutes.
   
   *Rationale:* Displaying coordination latency in real time creates positive pressure to sign promptly without compromising review quality.

5. **Byzantine resilience indicator:** For critical-consequence packets, a small shield icon indicates that Byzantine fault tolerance is active (from `engine/byzantine_resilience.py`). Hovering shows: `"3-of-4 quorum — no single ministry can authorise or block this action alone."`
   
   *Rationale:* Operators must understand that the quorum system is a security mechanism, not bureaucratic overhead.

#### Validation Rule Connection

`lib/packet/validate.ts` enforces that packets at `authorized` status or beyond must have `multiSig` with `currentSignatures >= threshold`. Packets that reach the console without meeting quorum are displayed with a red `"INSUFFICIENT SIGNATURES"` banner that blocks the "Execute" action.

## 3. Cross-Cutting Design Decisions

### 3.1 Industrial Dark Mode

**Palette:** Deep charcoals (#0f0f0f, #1a1a1a) with amber warnings (#f59e0b), cobalt active states (#3b82f6), and emerald validation (#10b981).

*Rationale:* Command-and-control environments typically use dark interfaces to reduce eye strain during extended monitoring and to make colour-coded alerts (amber, red) visually distinct. The palette avoids the clinical-white aesthetic of consumer applications, signalling to operators that this is operational infrastructure, not a dashboard toy.

### 3.2 Monospace Typography for Data

**Font:** JetBrains Mono for data values; Inter for prose.

*Rationale:* Monospace fonts ensure that numerical columns (correlation values, timestamps, node IDs) align correctly and are easily scannable. This is critical when operators are comparing evidence windows — misaligned decimals cause misreading under time pressure.

### 3.3 No Auto-Dismiss for Warnings

All validation warnings and uncertainty alerts persist on screen until explicitly acknowledged by the operator. There are no toast notifications that disappear after 5 seconds.

*Rationale:* Auto-dismissing warnings in crisis environments is dangerous. An operator who looks away for 5 seconds misses a critical warning. Persistent warnings require deliberate acknowledgement, creating a record of awareness.
